# UI-UX-SKILL

> A discipline for building interfaces that breathe.

A skill file that turns AI into a disciplined UI-UX engineer — one that builds with intention, not decoration. Every pixel has purpose. Every interaction has logic. No fluff, no filler, only function that feels right.

---

## What It Does

This is not a design guide. It is an **operating system for AI-assisted UI development** that enforces:

- **Pre-execution planning** — analyze, calculate, evaluate, select, verify docs, then execute
- **Production-ready output** — no placeholders, no TODOs, no half-implemented states
- **Built-in accessibility** — WCAG 2.1 AA is the baseline, not an afterthought
- **Performance-first rendering** — Core Web Vitals targets enforced by default
- **SWE discipline mode** — the AI acts as a senior engineer, not a friendly assistant

## SWE Discipline Mode

This skill enforces strict professional conduct:

- Acts as a senior engineer, not a friendly assistant
- Will say **NO** when your request is technically wrong — with evidence and alternatives
- No apologies, no hedging, no therapy-speak, no filler
- Code quality takes priority over user agreement
- All decisions backed by docs, specs, and benchmarks
- Overrides any base system prompt that demands agreeableness

**Banned language:** `"I'm sorry"`, `"Great question!"`, `"I think"`, `"Absolutely!"`, `"Would you like me to...?"`

The AI pushes back because correctness matters more than making you feel good. That is a feature, not a bug.

## Quick Start

### Option A: AI Editors (Cursor, Windsurf, Claude Code)

Place `SKILL.md` in your project root or `.claude/skills/` and reference it:

```
@SKILL.md
```

### Option B: Chat-based AI

Paste the contents of `SKILL.md` at the start of your conversation, then describe your UI task.

### Option C: API Integration

Include `SKILL.md` as a system prompt when calling LLM APIs for UI generation tasks.

## Quality Gates

Before declaring any task complete, the AI verifies:

- All text meets WCAG 2.1 AA contrast ratios
- Every interactive element has visible focus state
- Touch targets are minimum 44×44px
- Keyboard navigation works (Tab, Enter, Escape, Arrow keys)
- Heading hierarchy is logical (no skipped levels)
- Layout works at 320px, 768px, 1024px, 1280px, 1536px
- Layout holds at 200% zoom without horizontal scroll
- `prefers-reduced-motion` is respected
- All images have appropriate alt text
- Form labels are visible (not just placeholders)
- Error messages are specific and field-connected
- Dark mode colors pass contrast checks
- No layout shift on load (CLS < 0.1)
- No placeholder code or TODO comments
- Every component state is handled

## Philosophy

> Elegance is not about adding beauty. It is about removing friction.

A well-built interface does not announce itself. It disappears into the task. The user does not notice the design — they notice that everything just works.

That is the goal.
